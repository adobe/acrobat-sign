Hello World App¶

Once you’ve created a developer account, you’re ready to create your first app.

Create an app in the UI¶

To use Adobe Sign APIs, create an application.

  1. Log in to Adobe Sign.

  2. Select API from the top menu. If you are already an enterprise customer, you may not see the API link. In that case, choose Account.

  3. Select API Applications.

  1. Select the Create (+) icon and provide details about your app, including the intended use:

  • CUSTOMER:: Apps for internal use and testing.

  • PARTNER:: Select this type if you’re developing an application for other users.

Note

PARTNER applications will not have full access to other accounts until they have been certified. Click here to get your application certified.

Configure OAuth¶

OAuth is widely used by developers to authorize API access. Adobe Sign’s implementation complies with the OAuth 2.0 specification. Adobe Sign uses the OAuth authentication protocol to authorize requests for any Sign API endpoint. Get the Application ID and Application Secret from the web UI of Adobe Sign.

At a high level, the OAuth process requires the client application to request permissions from the end user before performing any actions on their behalf. This is accomplished by redirecting users to the Adobe Sign application where they will authenticate themselves and grant the requested permissions. Afterwards, the user is redirected back to the client web application.

The OAuth process returns Access Tokens which can be used to make REST API calls. Once you have created your application, click on the “Configure OAuth” link to configure your OAuth integration. This page contains the following information:

  • Your “Client Id” and “Client Secret”, which you will need during the token exchange.

  • You will need to specify the “Redirect URI”, which is the page on your website we’ll return users after the OAuth flow.

  • You will need to select which scopes you want to enable for your application.

Get the app ID and secret¶

  1. Select API Applications to view the list of apps you have created; then select your app to view its action menu.

  1. Select View/Edit to get the application ID and secret.

  2. Save the app’s application ID and secret. You’ll use this information to issue access tokens in the Adobe Sign API.

Set the app scopes¶

Scopes describe what resources and actions your application will access. Common scenarios include:

  • As a developer, you will need to enable the appropriate scopes for your application on the admin page.

    • This should be the minimal set of scopes necessary for your application to function.

    • If you apply for Certification, Adobe will review the enabled scopes to confirm that they match the intended application use.

  • Your application will make an OAuth authorization request with a scope parameter which indicates what permissions are being requested at that time.

    • The requested scopes must be a subset of the scopes that are enabled for the application.

    • The requested scopes must be appropriate for the action the user is attempting to perform.

    • The user will be asked to authorize the requested permissions for your application.

  • Assuming the OAuth tokens were successfully received, your application will make further API calls using the Access Token.

    • The calls that your application makes must be permitted by the scopes associated with the Access Token that’s being used.

    • For example, in order to call GET /agreements, the agreement_read scope must have been requested and authorized.

  1. Select Configure OAuth for Application to enable scopes.

Tip

The requested permissions on this screen sets the max allowable scopes, but every token is not going to exercise them all. Different tokens for various users in specific scenarios may use one or more of the the scopes (but not all).

Remember to limit the scopes that you intend to request through OAuth. The scope denotes the extent of permission that your users will have on your account. For example, agreement_send:account user_login:self would allow the application to send on behalf of any user in the account and also login on behalf of the user that authorized the request.

The following modifiers are available:

Modifier

Description

self

Perform the specified action on behalf of the authorizing user. This is the default: for example, “agreement_send:self” is the same scope as “agreement_send”

group

Perform the specified action on behalf of any user in the same group as the authorizing user. The authorizing user must be a group admin in order to grant this scope, and must have the Business or Enterprise edition of Adobe Sign.

account

Perform the specified action on behalf of any user in the same account as the authorizing user. The authorizing user must be an account admin in order to grant this scope, and must have the Business or Enterprise edition of Adobe Sign.

Now you can invoke the Sign APIs from your client-side apps.

OAuth token exchange¶

Authorization Request¶

The OAuth process starts with the client directing the user’s browser to make a request to the /public/oauth/v2 endpoint, with the following query string parameters:

Parameter

Value

Required?

Description

response_type

code

yes

The value must always be code

client_id

Your ID obtained from OAuth configuration page.

yes

Identifies the application

redirect_uri

A secure, absolute URI, for e.g. https://example.com.

yes

Redirects users here at the end of the authorization process. The value must belong to the set of values specified on the OAuth Configuration page.

scope

A space delimited set of permissions, specified during the OAuth configuration setup on the Configure OAuth page.

yes

The permissions that the user will be asked to approve

state

Any string

no

This value will be returned to the client as a parameter at the end of the authorization process. While not required, use of the state parameter is highly recommended to protect against CSRF as described in the RFC.

To invoke the Sign APIs, your application must first get an OAuth access token on behalf of an Adobe Sign user. This process uses HTTP requests to obtain the token. Your url will look like the following (with your client ID, etc.):

https://secure.adobesign.com/public/oauth/v2?
   response_type=code&
   client_id=CBAThIsIsNoTaReAlmPBvPF&
   scope=user_login:self+agreement_send:account&
   redirect_uri=https://myserver.com

Next, select Allow Access to receive the authorization code:

Once you select Allow Access, the page gets redirected to the following URL:

https://myserver.com/?
   code=CBNCKBAThIsIsNoTaReAlcs_sL4K32wCzs4N

Note the code CBNCKBAThIsIsNoTaReAlcs_sL4K32wCzs4N in the request. This is the authorization code that you will use to get the OAuth access token.

You can send the authorization code along with the client ID and client secret to the Sign service to get the token. Use the same api_access_point that you got from the previous response.

POST https://api.adobesign.com/oauth/v2/token?
   grant_type=authorization_code&
   code=CBNCKBAThIsIsNoTaReAlcs_sL4K32wCzs4N&
   client_id=CBAThIsIsNoTaReAlmPBvPF&
   client_secret=319UThIsIsNoTaReAl2-4OxkVo9ycU&
   redirect_uri=https://myserver.com HTTP/1.1

Content-Type: application/x-www-form-urlencoded

You will get the following JSON body containing the access token and the refresh token:

{
   "access_token":"3AAABLblThIsIsNoTaReAlToKeNPr6Cv8KcZ9p7E93k2Tf",
   "refresh_token":"3AAABLblThIsIsNoTaReAlToKeNWsLa2ZBVpD0uc*",
   "token_type":"Bearer",
   "expires_in":3600,
   "api_access_point ":"https://api.na1.adobesign.com/",
   "web_access_point":" https://secure.na1.adobesign.com/"
}

Note the following tokens in the response body:

  • Access Token:: Use this token to access any Adobe Sign API endpoint.

  • Refresh Token: If your access token expires, use the refresh token to request for a new access token. Keep your Client ID and Client Secret handy to request for a new access token from a refresh token.
